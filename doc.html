用户守则

Cloud I Wanna 用户须遵守以下规则：

  • 禁止任何方式的游戏作弊，包括但不限于代码修改、变速工具、按键脚本、一键JC
  • 禁止恶意刷分、恶意举报
  • 禁止涉及违规的言论、恶意言论、人身攻击
  • 禁止发布无意义、恶意高难的游戏，游戏内禁止出现违规内容
  • 禁止任何方式的爬取、篡改数据库、修改应用程序

    • 注：违规内容包括但不限于涉及政治、广告、引战、谣言、色情、血腥暴力、猎奇的内容

    应用相关

    储存

    Cloud I Wanna 大量使用本地储存以提高加载速度和支持离线功能，具体储存内容如下：

  • 基础储存：代码缓存，应用数据等
  • 用户储存：用户头像，消息记录
  • 游戏储存：游戏列表，游戏数据
  • 资源储存：游戏内图像、音效、字体、背景音乐
  • 其它储存：玩家存档，制作项目

    • 当储存占用空间过大时，可在设置中清理应用缓存 (该操作不会影响离线使用)，或者选择清理相应的储存数据 (该操作可能会影响离线使用)

    网络

    CDN

    应用数据、自动更新、游戏字体的下载会使用CDN，不同的CDN节点速度和稳定性会有所不同。如遇上述内容加载很慢甚至加载错误的情况，可在应用设置中尝试更换CDN节点。

    离线使用

    考虑到可能会发生无法连接到服务器的情况，Cloud I Wanna 支持离线使用。离线情况下，游戏和制作功能都能正常使用，但是只能访问已储存到本地的资源。

    版本更新

    Cloud I Wanna 应用程序的版本更新和内容的更新是分离的。在主页左下角可以看到版本信息：v0.x.x build.xxxxx，前者是应用程序版本，后者是内容版本。内容更新频率很高，会在每次打开 Cloud I Wanna 时自动完成，而应用程序只会在一些底层接口变更时更新。应用程序版本更新时，会提示用户重启完成自动更新；跨大版本更新时，需要用户前往官网手动下载更新。

    游戏相关

    引擎特性

    Cloud I Wanna 游戏引擎使用原生 JavaScript 开发，完全还原常规 I Wanna 操作手感，同时修复了由 GameMaker 引擎引起的bug。具体来说，与常用GMS引擎的细微差别如下：

  • 游戏引擎使用双精度浮点 (GMS是单精度浮点)，且碰撞检测的舍入规则采用一般的四舍五入 (相比之下GMS是银行家舍入)，因此利用 0.5align 相关bug的特性在 Cloud I Wanna 中失效，部分 y-align 的表现也会与GMS引擎不同

    • 游戏存档

    游戏联机

    联机功能

    Cloud I Wanna 提供了方便的游戏联机功能

    联机模式

    普通模式：仅显示玩家

    竞技模式：竞技将强制所有玩家同时从头开始，系统会推送玩家的游戏进度信息

    合作模式：玩家间将共享游戏数据，包括存档位置、全局变量、联机触发器和计数器等。合作模式下，第一次按R回到服务器最新存档，再次按R回到自己本地的存档，按Ctrl+R强制回到服务器最新存档。合作模式下存档是临时的，不会录像，通关记录也不会上传

    制作相关

    基本使用

    游戏房间

    游戏房间分为四类：

  • 初始房间：游戏最初的房间
  • 通关房间：到达此房间，游戏即通关，计时会停止，同时上传通关记录
  • 普通房间：计入游戏流程的房间，会存档、计时，属于游戏流程的应当都使用普通房间
  • 过场房间：不属于游戏流程的房间，不会存档、计时，可用做游戏开始前的各类房间

    • 在使用时应当注意：

  • 在开始游戏(或读档)前不应出现普通房间，这会干扰存档的正常工作
  • 初始房间是游戏的入口，并不一定要做成标题界面，例如可以将初始房间做成音量检查，然后在后续的过场房间做标题
  • 过场房间的用途很灵活，可用于读档界面、难度选择、开场动画、游戏ED等

    • 图层

    Cloud I Wanna 游戏引擎共有12个图层，对应深度0~11，图层0在最底层，图层11在最顶层。其中图层7是玩家所在图层；图层10和11可设置为GUI图层，GUI图层物体的绘制相对于游戏窗口位置不变。需要注意的是，由于游戏运行会执行优化，同一层内物体的摆放顺序并不一定会决定最终绘制的前后。

    在工具栏点击图层按钮可展开图层控制面板，每个图层有3个控制按钮，分别对应“单独显示”、“隐藏”和“锁定”。开启“单独显示”后，其余图层会自动隐藏，处于隐藏或锁定状态下的图层无法进行物体摆放、选中、删除操作。

    运行与调试

    在编辑器中运行游戏，可以选择从当前房间开始、从头开始、从存档继续三种模式，当选择从当前房间开始时，每次运行存档将被覆盖。在编辑器中运行游戏时，存档是临时的，即离开当前项目后存档失效，因此若要完整地测试存档，请在工具栏[测试]中选择[独立测试]。

    游戏运行中，状态栏会显示当前逻辑帧率、逻辑占用、绘制帧率和绘制占用。不同于 GameMaker，Cloud I Wanna 的逻辑帧与绘制帧是分离的，逻辑部分按照每秒50帧恒定运行，而绘制仅当画面需要更新时才发生。因此正常运行时，逻辑帧率应当恒定在50，而绘制帧率则有可能低于50。逻辑占用和绘制占用是根据帧生成时间和绘制时间计算的估计值，当逻辑占用高于50%时，需要检查物体的逻辑是否正确，并尝试优化；当绘制占用高于50%时，需要减少粒子等特效的数量。

    物体属性

    物体属性面板用于编辑选中物体。属性 (或动作) 支持多选 (按住Ctrl)、连续多选 (按住Shift)、全选 (Ctrl+A) 操作，在属性的右键菜单里，可以进行删除 (Delete或退格)、复制 (Ctrl+C)、粘贴 (Ctrl+V)、剪切 (Ctrl+X) 等操作，除此之外，在添加属性处右键可以粘贴复制的属性。对于时间轴和动作组里的动作，拖动动作前的图标可以更改动作的顺序。需要注意的是，同时选中多个物体时，显示和编辑的也只是首个物体的属性，若要将当前修改应用到所有选中物体，需要在右键菜单 (或面板右侧的“”按钮展开菜单) 中点击[全部应用]。

    时间轴

    时间轴用以实现按时序执行的一系列动作，其运行逻辑如下：

  • 从上到下依次执行每个节点 (注意是在同一帧内发生的，没有时间间隔！)
  • 遇到[间隔]，等待指定的帧数后再继续向下
  • 遇到[循环]，跳转到指定的节点 (注意同样没有时间间隔！)

    • 开启[直接运行]时，时间轴会在物体创建后自动运行，关闭后则可以通过[开始时间轴]等动作手动控制其执行。时间轴的间隔除了指定帧数，还可以设置成节拍数 (这在制作耐久时非常有用！)。默认情况下，1拍=50帧，而这个值可以通过[设置时间轴节拍]动作修改。在设置时间轴的循环节点时，需要注意以下事项：

  • 在到达循环节点前，动作已经执行了一次，因此虽然循环次数为n，但循环节点之间的动作实际上执行了n+1次
  • 当循环次数设置为无限时，若循环节点之间没有时间间隔，将会造成死循环

    • 动作组

    动作组用于实现在游戏运行的特定节点执行一系列动作。在设置动作组时，需要根据实际情况确定其执行的位置。设置为“创建时执行”时，该动作组会在物体创建时执行一次；设置为“销毁时执行”时，该动作组会在物体销毁时执行一次；设置为“手动执行”时，该动作组不会自动执行，需要通过[执行动作组]动作调用执行。设置为其余选项时，该动作组会每帧都执行，引擎每帧内各个节点的执行顺序如下：

    [时间轴前]动作组执行所有物体的“时间轴前执行”动作组
    时间轴执行所有物体的时间轴
    玩家操作执行玩家的操作逻辑
    [移动前]动作组执行所有物体的“移动前执行”动作组
    物体移动物体按照设置的运动移动到新位置
    玩家移动玩家移动到新位置
    [移动后]动作组执行所有物体的“移动后执行”动作组
    玩家碰撞处理执行玩家碰撞致命物体、传送门、存档等判定
    物体同步有同步位置动作的物体移动到新位置

    通常情况下，动作组设置为“移动前执行”即可；触发器的开关动作可设置在“时间轴前执行”，以便同一帧内其他物体可以在[移动前]动作组检测到触发器的开关。

    发布到游戏分区

    游戏分区是发布完整游戏的分区。在这一分区发布的是“游戏”而不是“关卡”，也就是说和使用 GameMaker 制作的 I Wanna 一致。游戏分区完整地支持存档管理、进度结局、成就系统、速通计时器功能，还可以开启独立窗口模式带来原生游戏体验。

    不同流程、难度的游戏，玩家通关获得的经验理应不同，因此在发布游戏时需要设置游戏经验。由于经验难以精细量化，Cloud I Wanna 划分了7个档位的游戏经验，这里给出一些评估参考：

  • 3经验游戏：通常是只有少数几面的游戏，理论上几分钟就可以通关。例如：kokomi
  • 10经验游戏：流程较短，理论上十几分钟、半小时左右就可以通关，一般为线性流程。例如：I wanna Uhuhu Spike
  • 20经验游戏：理论上可能需要一个小时，甚至更长的时间通关，一般都是分为几个小关，每个小关流程不至于太长。例如：I wanna run the Marathon
  • 40经验游戏：理论上需要几个小时通关的，一般经典综合大作、较难的百层跳刺都属于这个范畴。例如：Not Another Needle Game, I wanna see the moon
  • 80经验游戏：需要几十个小时通关的游戏。例如：Not Another Magic Tower Game, Crimson Needle 3

    • 以上仅供参考，请根据实际情况进行评估。经验设置错误也不必担心，可以通过管理员做出调整。

    更新游戏

    当需要对已发布游戏做出修改时 (如bug修复、特效更新等)，可以使用游戏更新功能。需要注意的是，更新游戏需要保证游戏存档的兼容性，即当前更新不影响旧版本的已有流程，否则会导致存档错误或录像错误。如果对旧版本流程有重大修改，建议重新发布游戏，而不是使用游戏更新。

    游戏更新还可用于发布未完成作品或者连载作品，具体参考以下流程：

  • 当前版本的关卡结束后，在下次要更新关卡的开始位置自动存档
  • 玩家到达下次要更新关卡的开始位置并自动存档后，传送到通关界面
  • 通关界面不放置玩家和存档 (到达通关房间之后不能以任何方式存档)，仅用于上传通关记录
  • 发布游戏时，按照现有游戏流程设置经验
  • 下次更新游戏时，勾选[删除旧版通关记录] (此操作并不会删除玩家存档)

    • 按照上述流程操作，就能在保留未通关状态存档的同时上传当前版本的通关记录。虽然按照上述流程操作可以实现连载作品，但在游戏连载的过程中难免会对已有的关卡也做出修改，使得旧存档失效。因此，我们并不建议发布未完成作品或者连载作品。

    发布到关卡分区

    关卡分区是为演示、测试、教程、素材等单房间关卡设计的分区，本质上作为游戏分区的辅助，旨在促进作者与作者、作者与玩家的有效交流。完整游戏的制作通常需要花很多时间和精力，而关卡制作简单快速，适合入门学习和积累经验。在这个分区，你可以练习制作、进行教学、测试功能和想法、发布素材，当然也可以制作短图换换风格！

    关卡分区带有临时性质和快餐性质，容易出现低创内容，因此系统会进行定期清理。当一个关卡一段时间都没有通关数据更新时，清理策略如下：

    评分 ≥ 8.0永久保留
    评分 ≥ 7.090天无更新
    评分 ≥ 6.030天无更新
    评分 < 6.015天无更新
    评分 < 4.07天无更新
    评分 < 2.03天无更新
    无评分15天无更新

    注意，只有关卡分区会定期清理，游戏分区是不会的哦！

    导出游戏

    Cloud I Wanna 导出游戏目前支持网页游戏和 Windows 独立游戏。网页游戏导出类型为 HTML5 静态网页，其文件结构如下：

    导出目录
 ├─data
 │  ├─font 游戏字体
 │  ├─music游戏音乐
 │  ├─sound游戏音效
 │  └─[Game ID] 独立的游戏数据目录
 │     ├─icon   游戏图标
 │     ├─game   游戏数据 (json)
 │     ├─texture纹理图像 (webp/png)
 │     └─uv     纹理uv数据
 ├─css引擎所需CSS文件
 ├─js 引擎所需JS文件
 └─index.html游戏页面

    导出网页游戏会为每个游戏单独创建名为游戏ID的数据目录，因此可以将多个游戏导出到同一目录下，通过自定义index.html中的GID变量进行路由。需要注意的是，网页游戏并不能直接双击index.html运行，而是需要部署到静态网页服务器，要在本地调试，可以在导出时勾选“包含本地调试程序”。

    Windows 独立游戏导出类型为64位可执行程序，其文件结构如下：

    导出目录
 ├─data
 │  ├─font 游戏字体
 │  ├─music游戏音乐
 │  ├─sound游戏音效
 │  ├─icon   游戏图标
 │  ├─game   游戏数据
 │  ├─texture纹理图像
 │  └─uv     纹理uv数据
 └─[Game Name].exe可执行程序

    Windows 独立游戏依赖 Microsoft Edge WebView2 运行环境 (在 Windows 10 及以上版本系统自带)，若提示缺少运行环境，可前往微软官方下载。

    注意事项

    性能优化

    Cloud I Wanna 游戏引擎使用原生 JavaScript 开发，非编译的特性使得其运行速度低于编译程序，因此在开发时做了大量优化工作以提高游戏性能。虽然如此，由于游戏制作功能复杂，不正确的操作会极大地降低游戏性能。游戏运行速度和物体数量成反比，耗时最多的部分是碰撞检测与处理，且旋转会进一步降低其运行速度，除此之外，不恰当的特效也会影响游戏性能。这里给出一些性能优化建议：

  • 避免大房间，使用多个小房间来代替，同时减少不必要的物体使用
  • 减少碰撞检测的使用，将碰撞逻辑放在物体数量少的一方
  • 设置条件执行的条件时，碰撞检测作为最后的条件 (如使用“[可见]且[碰撞固体]”而不是“[碰撞固体]且[可见]”)
  • 尽量使用固定视野而不是跟随视野
  • 尽量减少物体的旋转
  • 弹幕、粒子请使用经过优化的特定动作，而不是使用普通物体实现
